У нас критична і глобальна проблема з архітектурою завантаження даних (Data Fetching) у Next.js App Router. Дані періодично зависають (вічний спінер), проміси вмирають без помилок, а при клієнтському роутингу (зміна дат, перехід між сторінками) дані або грузяться по 10 секунд, або взагалі крутить вічний спінер. Лікується тільки хард-рефрешем сторінки. Як тільки заходиш на сайт - все рівно, але якщо він повисів хвилин 5 на фоні - все, піздєц. Та і загалом, навіть якщо відкритий - ось сторінка клієнтів відкрилась, далі перевірив записи, знову на клієнтів - а вона вже не грузить.
Твоя задача — провести повний аудит ВСІЄЇ логіки фетчингу в проєкті, скласти план і повністю переписати цей шар з нуля за єдиним правильним і жорстким стандартом.

ВЕКТОР ДІЙ (Режим Планування):

Глобальний скан: Проаналізуй src/app, src/components та src/lib/supabase. Знайди всі місця, де відбувається отримання даних (клієнтські та серверні компоненти).

Аналіз проблеми: Вияви всі Server Actions, які використовуються для фетчингу всередині клієнтських компонентів (це Root Cause зависань). Вияви всі хуки React Query, які неправильно обробляють стан завантаження або втрачають інстанс Supabase.

Генерація плану: Напиши покроковий план рефакторингу. НЕ ПОЧИНАЙ писати код, поки Користувач не затвердить план. План має розбити рефакторинг на логічні модулі (наприклад: 1. Supabase Client, 2. Server Fetching, 3. Client Hooks, 4. UI/UX Smoothness).

ЖОРСТКІ АРХІТЕКТУРНІ ПРАВИЛА (ЯК МАЄ БУТИ ПІСЛЯ РЕФАКТОРИНГУ):

Правило 1 (Server Components): Фетчать дані напряму через createClient (server client) без React Query.

Правило 2 (Client Components & React Query): Використовують ТІЛЬКИ createBrowserClient(). Цей клієнт має бути синглтоном або жити в стейті провайдера, щоб не перестворюватись на кожен рендер.

Правило 3 (СМЕРТЬ SERVER ACTIONS ДЛЯ ФЕТЧИНГУ): Server Actions використовуються ВИКЛЮЧНО для мутацій (POST, PUT, DELETE). Жоден клієнтський компонент чи useQuery не має викликати Server Action для отримання даних (GET). Тільки прямі запити supabase.from... через Browser Client.

Правило 4 (Захист від зависань): Всі queryFn у React Query повинні мати жорстку обробку помилок (if (error) throw error). Жодних "тихих" фейлів.

Правило 5 (Миттєвий UI): Всі списки, календарі та статистика у React Query мусять мати placeholderData: keepPreviousData (або еквівалент для v5). UI ніколи не має показувати голий loading-екран при зміні параметрів чи дат — тільки фонове оновлення (isFetching).
